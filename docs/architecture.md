# Tinyscript Architecture

Developer-focused internals guide. For language features and usage examples, see the top-level `README.md`.

---

## 1. Project Overview

Tinyscript is a proof-of-concept JavaScript-like scripting language built on [Eclipse Xtext](https://eclipse.dev/Xtext/). It demonstrates how Xtext can be used to build a complete, executable language toolchain: grammar → parser → AST → interpreter, with IDE integration coming for free.

**Goals:**
- Educational exploration of language engineering with Xtext/EMF
- Sandboxed execution suitable for SaaS multi-tenant scenarios (user-submitted scripts run safely)
- Study of prototypal inheritance, closures, and dynamic typing in a minimal implementation

**Deployment modes:**
- **Eclipse IDE plugin** — syntax highlighting, auto-complete, validation in the Eclipse editor
- **Standalone REPL** — interactive command-line shell
- **Spring Boot web demo** — sandboxed script execution via REST API + browser playground
- **JVM ScriptEngine** — compiles Tinyscript to JavaScript, then executes via GraalJS/Nashorn

## 3. Execution Pipeline

```
Source text (.ts file or inline string)
  │
  ▼
Xtext parser
  (generated from Tinyscript.xtext via GenerateTinyscript.mwe2)
  │
  ▼
EMF AST (EObject tree)
  (model classes in src-gen/de/mkbauer/tinyscript/ts/)
  │
  ▼
TinyscriptValidator
  (static semantic checks: duplicates, illegal return, reserved names)
  │
  ▼
TinyscriptEngine                    ← main interpreter
  (tree-walking evaluator, ~964 lines)
  │
  ▼
TSValue result
```

The parser and AST model classes are entirely generated — never edit the `src-gen/` directory by hand. The validator and interpreter are hand-written.

## 4. Grammar & Parser (`Tinyscript.xtext`)

The Xtext grammar file is the single source of truth for the language syntax. From it, Xtext generates:
- An ANTLR-based parser
- EMF model classes for every grammar rule (the AST node types)
- A language server (LSP) implementation
- Scope-resolution infrastructure hooks

**Key construct families in the grammar:**

| Family | Grammar rules |
|---|---|
| Top-level | `Tinyscript`, `Statement` |
| Statements | `VariableStatement`, `FunctionDeclaration`, `ReturnStatement`, `IfStatement`, `ForStatement`, `WhileStatement`, `Block` |
| Expression precedence chain | `AssignmentExpression` → `ConditionalExpression` → `OrExpression` → `AndExpression` → `EqualityExpression` → `RelationalExpression` → `AdditiveExpression` → `MultiplicativeExpression` → `UnaryExpression` → `PostfixExpression` → `CallOrPropertyAccess` → `PrimaryExpression` |
| Literals / initializers | `NumericLiteral`, `StringLiteral`, `BooleanLiteral`, `NullLiteral`, `ObjectInitializer`, `ArrayInitializer`, `FunctionExpression` |
| Property/call chain | `CallOrPropertyAccess` with `arguments` or `property` suffixes |

**Generated output location:** `de.mkbauer.tinyscript/src-gen/`

**Regenerating after grammar changes:**
```
Run: de.mkbauer.tinyscript/src/de/mkbauer/tinyscript/GenerateTinyscript.mwe2
(right-click → Run As → MWE2 Workflow in Eclipse, or via Maven)
```

## 5. Interpreter: `TinyscriptEngine`

**File:** `interpreter/TinyscriptEngine.java` (~964 lines)

The interpreter is a classic tree-walking evaluator. It recursively visits EMF AST nodes and computes a `TSValue` result for each.

### Dispatch

The single universal entry point is:
```java
public TSValue execute(EObject object)
```
It dispatches via an integer `switch` on the EMF classifier ID:
```java
switch (object.eClass().getClassifierID()) {
    case TsPackage.BINARY_EXPRESSION:
        return caseBinaryExpression((BinaryExpression) object);
    case TsPackage.BLOCK:
        return caseBlock((Block) object);
    case TsPackage.CALL_OR_PROPERTY_ACCESS:
        return caseCallOrPropertyAccess((CallOrPropertyAccess) object);
    // ... 24 cases total
    default:
        throw new UnsupportedOperationException(...);
}
```
This approach is faster than `instanceof` chains and avoids reflection — the EMF classifier IDs are compile-time integer constants in `TsPackage`.

### Key State Fields

```java
private ExecutionContext         currentContext;       // active variable scope
private GlobalExecutionContext   globalContext;        // root scope (global object)
private Deque<ExecutionContext>  contextStack;         // scope stack (ArrayDeque)
private Map<Block, LexicalEnvironment> lexicalEnvironments; // hoisted declarations per block
private TSObject                 objectPrototype;      // Object.prototype singleton
private ResourceMonitor          resourceMonitor;      // enforces sandbox limits
private boolean                  sandboxed = true;     // disables fs and other privileged built-ins
protected int                    callDepth;            // current call stack depth
private OutputStream             stdOut;               // where print() writes
```

### Context Stack Management

```java
enterNewExecutionContext(String name)
enterNewExecutionContext(String name, ExecutionContext outer)
leaveExecutionContext()
```
`enterNewExecutionContext` pushes `currentContext` onto `contextStack` and creates a new child `ExecutionContext`. `leaveExecutionContext` pops it back. This is called on every function call and for block-scoped constructs.

### Function Hoisting

Before executing a `Block`, `TinyscriptEngine` pre-scans it for `FunctionDeclaration` nodes (via `getLexicalEnvironment(block)`). These are stored in a `LexicalEnvironment` and declared in the current context before any statements run — matching JavaScript's function hoisting behavior.

### Exception-Based Return Values

`return` statements throw a `TSReturnValue` (an unchecked exception wrapping a `TSValue`). This unwinds the call stack back to `InterpretedFunction.apply()`, which catches it and extracts the return value. No special control-flow mechanism is needed.

### Top-Level Entry

`caseTinyscript(Tinyscript object)` is the script entry point. It resets `contextStack` and `callDepth`, sets up the global context, calls `resourceMonitor.start()`, hoists top-level function declarations, then executes all statements.

## 6. Type System: `TSValue` and `TSObject`

### `TSValue`

**File:** `interpreter/TSValue.java` (191 lines)

All runtime values are wrapped in a `TSValue`. Internally it holds a single `Object value` field — a boxed Java type:

| Tinyscript type | Java storage |
|---|---|
| number | `Double` (always normalized from any `Number`) |
| string (primitive) | `String` |
| string (object) | `StringObject` |
| boolean | `Boolean` |
| object / array / function | `TSObject` (or subclass) |
| `undefined` | `null` (identity = `TSValue.UNDEFINED`) |
| `null` | `null` (identity = `TSValue.NULL`) |
| `NaN` | `Double.NaN` (singleton = `TSValue.NAN`) |

`UNDEFINED` and `NULL` are distinguished purely by Java object identity (`this == TSValue.UNDEFINED`), not by the `value` field content.

**Type query methods:** `isString()`, `isDouble()`, `isNumber()`, `isBoolean()`, `isObject()`, `isArray()`, `isCallable()`, `isInfinity()`, `isObjectCoercible()`

**Coercion methods:** `asString()`, `asDouble()`, `asInt()`, `asBoolean()`, `asObject()`, `typeOf()`

`typeOf()` returns `"object"` for `NULL`, matching the ECMAScript specification.

### `TSObject`

**File:** `interpreter/TSObject.java`

`TSObject` is the backing store for all objects, arrays, and functions.

```java
protected HashMap<String, TSPropertyDescriptor> properties;
private   TSObject proto;   // prototype chain pointer
```

Each `TSPropertyDescriptor` carries:
- `TSValue value`
- `boolean configurable`, `writable`, `enumerable` (all default to `true`)

**Prototype chain lookup:**
```java
public TSValue get(String key) {
    TSPropertyDescriptor desc = properties.get(key);
    if (desc != null) return desc.getValue();
    if (proto != null) return proto.get(key);   // recursive walk
    return TSValue.UNDEFINED;
}
```

Writes (`put`) always go to the object's own `properties` map, never up the prototype chain.

**Setting the prototype:**
```java
setPrototype(TSObject proto)
```
This stores the proto reference both in the `proto` field and as a regular `"__proto__"` property, so prototype inspection works from script code.

**Static utility methods on `TSObject`:** `toString()`, `toObject()`, `toPrimitive()`, `toNumber()`, `toInteger()` — these implement the ECMAScript abstract coercion operations.

**Memory estimation:** `estimateMemory()` returns `properties.size() * 8` bytes (TSVALUE_SIZE = 8). Used by the resource monitor.

## 7. Scoping & Closures: `ExecutionContext`

### `ExecutionContext`

**File:** `interpreter/ExecutionContext.java`

Each function call and block creates a new `ExecutionContext` (an activation record / stack frame):

```java
private HashMap<String, TSValue> values;   // variable bindings for this frame
private ExecutionContext          outer;    // parent scope (closure chain)
protected TSObject               thisRef;  // `this` value for this frame
protected boolean                functionContext; // true = function boundary
```

**Three core scope operations:**

| Operation | Method | Behavior |
|---|---|---|
| Declare | `create(id)` | Writes `UNDEFINED` into local `values`; throws on duplicate |
| Read | `lookup(id)` | Walks the `outer` chain; throws `TinyscriptReferenceError` if not found at global level |
| Write | `store(id, value)` | Writes into the frame where `id` was declared (walks chain); throws at global if undeclared |

`get(id)` is a local-only check (no chain walk) — used internally to avoid infinite recursion.

### `GlobalExecutionContext`

**File:** `interpreter/GlobalExecutionContext.java`

The root scope, extends `ExecutionContext` with `outer = null`. All global variables live as properties on a `TSObject globalObject`:

```java
private TSObject globalObject;
```

All scope operations delegate to `globalObject.get/put`. Unlike regular `ExecutionContext.lookup()`, the global version does **not** throw on missing identifiers — it returns `TSValue.UNDEFINED` (via `TSObject.get()`). `thisRef` is set to `globalObject`, so `this` at global scope refers to the global object.

### Closures

Closures are implemented by `InterpretedFunction` capturing the `ExecutionContext` that was active at function-definition time:

```java
private ExecutionContext outerContext;   // captured at definition time
```

When the function is called, `enterNewExecutionContext(name, outerContext)` creates a new frame whose `outer` pointer is the **captured** context, not the call-site context. This gives correct lexical scoping for closures.

**Arrow functions** additionally inherit `thisRef` from `outerContext.getThisRef()` instead of from the call-site `self`, implementing lexical `this`.

## 8. Function Types

```
TSObject
└── Function  (abstract)
    ├── InterpretedFunction  — user-defined; holds AST block + captured outerContext
    ├── BuiltinFunction      — native Java lambda via BuiltinFunctionImplementation
    └── BuiltinConstructor   — abstract; subclasses implement createObject()
          ├── ObjectConstructor
          ├── ArrayConstructor
          ├── StringConstructor
          └── FunctionConstructor
```

All function types are `TSObject` subclasses, so functions are first-class objects and can have properties set on them.

**`Function` (abstract base):**
```java
public abstract TSValue apply(TSObject self, TSValue[] args);
public abstract String getName();
public abstract int getLength();           // declared parameter count
public boolean hasInstance(TSValue value)  // implements instanceof via prototype chain walk
private boolean isArrowFunction;
```

**`InterpretedFunction.apply()` call flow:**
1. `resourceMonitor.checkMXCpuTimeAndMemory()` — CPU/memory check before entering
2. `engine.enterNewExecutionContext(name, outerContext)` — new frame, parent = captured closure scope
3. Increment `engine.callDepth`; `monitor.checkCallDepth()`
4. Set `currentContext.thisRef = self` (or `outerContext.thisRef` for arrow functions)
5. Set `currentContext.functionContext = true`
6. Bind positional parameters; collect rest args into `ArrayObject`
7. Hoist nested function declarations via `engine.getLexicalEnvironment(block)`
8. Execute `engine.caseBlock(block)`; catch `TSReturnValue` for explicit returns
9. `engine.leaveExecutionContext()` in `finally` — always restores previous scope

**`BuiltinFunction.apply()`:** Validates argument count, then calls `implementation.apply(self, args)` where `implementation` is a `@FunctionalInterface`.

**`BuiltinConstructor`:** Concrete subclasses implement `createObject()` to allocate the new object. `apply()` returns `new TSValue(self)` (the newly created object that `new` passes as `self`).

## 9. Built-in Runtime Library

Built-ins are registered in `TinyscriptEngine.initialize()` and also declared in a synthetic `__builtins.ts` Xtext resource (so the IDE scope provider can resolve them for auto-complete).

| Object | Class | Key methods |
|---|---|---|
| `Array` | `ArrayConstructor`, `ArrayObject` | `push`, `pop`, `shift`, `unshift`, `map`, `filter`, `join`, `concat`, `toString`, `Array.isArray()` |
| `String` | `StringConstructor`, `StringObject` | `charAt`, `indexOf`, `substring`, `toString` |
| `Math` | `MathObject` | `PI`, `sqrt`, `round`, `random` |
| `Object` | `ObjectConstructor` | `create`, `keys`, `getPrototypeOf`, `defineProperty`, `getOwnPropertyDescriptor`, `getOwnPropertyNames`, `hasOwnProperty` |
| `Function` | `FunctionConstructor` | `call`, `apply`, `toString` |
| `fs` | `FSObject` | `readFile`, `writeFile` _(disabled when sandboxed)_ |
| `System` | `SystemObject` | `currentTimeMillis` |
| `print` | `Print` | global function |

Each method is a `BuiltinFunction` whose `implementation` field is a Java lambda. Array and String prototype methods are attached to the shared prototype object of their constructor, so all instances inherit them via the prototype chain.

## 10. Sandboxing & Resource Monitoring

### `ResourceMonitor`

**File:** `interpreter/ResourceMonitor.java`

Tracks multiple resource dimensions and throws `TinyscriptResourceLimitViolation` (unchecked) when a limit is exceeded.

**Tracked dimensions and when they are checked:**

| Event hook | Checked limit(s) |
|---|---|
| `recordStatement()` | `maxStatements` |
| `checkCallDepth(int depth)` | `maxCallDepth` |
| `recordObjectCreation(TSObject)` | `maxObjectCreations`, `maxObjectSize`, `maxObjects`, `maxMemory` |
| `recordObjectSizeChange(TSObject)` | `maxObjectSize`, `maxObjects`, `maxMemory` |
| `recordStringCreation(String)` | `maxStringLength`, `maxObjects`, `maxMemory` |
| `checkMXCpuTimeAndMemory()` | `maxMXCpuTime` (thread CPU ns), `maxMxMAlloc` (thread allocated bytes) |

**Object tracking** (`enableObjectTracking()`): Uses a `WeakHashMapWithCallBack<Object, Integer>` to track live `TSObject` and `String` instances. When an object is GC'd, the callback decrements `totalMemory` automatically. Memory is estimated as `properties.size() * 8` bytes per object and `length * 2` bytes per string.

**MX Bean inspection** (`enableMXBeanInspection()`): Uses `ThreadMXBean` for per-thread CPU time and `com.sun.management.ThreadMXBean` for per-thread allocated bytes — both measured relative to thread start to isolate per-execution costs.

### `ResourceLimits`

**File:** `interpreter/ResourceLimits.java`

Plain value object carrying all limit thresholds. Zero = unlimited. `ResourceLimits.UNLIMITED` is a constant with all zeros.

```java
int  maxStatements;
int  maxCallDepth;
int  maxObjects;
long maxMemory;          // bytes
long maxMXCpuTime;       // nanoseconds
long maxMxMAlloc;        // bytes
int  maxObjectCreations;
int  maxObjectSize;      // max properties per single object
int  maxStringLength;
```

### Web Demo Limits

Set by `TinyscriptExecutionService`:

| Limit | Value |
|---|---|
| `maxStatements` | 5,000,000 |
| `maxCallDepth` | 128 |
| `maxMemory` | 16 MB |
| Max output size | 8 KB |

### Sandboxed Flag

`engine.sandboxed = true` disables the `fs` built-in and any other privileged operations. The web demo always runs with sandboxing on; the REPL runs with it off.

## 11. Validation & Scoping (Static Analysis)

### `TinyscriptValidator`

**File:** `validation/TinyscriptValidator.java` (81 lines)

Xtext calls `@Check`-annotated methods during parsing and on-save in the IDE. Violations become error markers (red squiggles).

| Check method | What it validates |
|---|---|
| `checkVariableDoesNotCauseConflictInBlock` | No two `var` declarations with the same name in the same `Block` |
| `checkFunctionDeclarationDoesNotCauseConflictInBlock` | No `function` declaration name conflicts in the same `Block` |
| `checkReservedIdentifier` | Identifiers must not start with `"__ts"` (reserved for transpiler shims) |
| `checkReturnInvalidInGlobal` | `return` statement not allowed at global (top-level) scope |

These checks are compile-time only — the interpreter does not re-run them at runtime.

### `TinyscriptScopeProvider`

**File:** `scoping/TinyscriptScopeProvider.java`

Builds lexical scope chains for the IDE's cross-reference resolution and auto-complete. This has no effect on runtime behavior — the interpreter uses `ExecutionContext` for scope, not the Xtext scope provider.

## 12. Code Generator (Tinyscript → JavaScript)

**File:** `generator/TinyscriptGenerator.xtend` (compiled to `xtend-gen/...TinyscriptGenerator.java`)

Transpiles the Tinyscript AST to ES5-compatible JavaScript source text.

**Used by:**
- `/xcompile` REST endpoint (`TinyscriptCrossCompilingService`)
- `JvmScriptEngineRunner` — compiles to JS then runs via GraalJS or Nashorn

**Not used by the interpreter** — the interpreter walks the AST directly without a JavaScript intermediate.

**Tested by:** `de.mkbauer.tinyscript.generator.tests` — a plain-Maven module (no OSGi) that verifies the generator differentially. See §15.

### Dispatch

The generator uses a long `instanceof` chain in the `generate(EObject)` dispatcher (this is how Xtend compiles its dispatch methods). Unlike the interpreter's integer switch, this is slightly slower but the generator is not performance-critical.

### Runtime Shims (`generateBuiltins()`)

The generator prepends small JavaScript shims to make the transpiled output self-contained:

```javascript
// Tinyscript assert() → JS function
function assert(condition) { if (!condition) throw "AssertionError"; }

// System.currentTimeMillis() → Date.now()
var System = {}; System.currentTimeMillis = Date.now;

// print() polyfill for environments without it (Node.js has console.log, not print)
if (typeof print !== "function") { print = function(s) { console.log(s); }; }

// __ts_add: handles Tinyscript's array+ semantics
// The + operator in Tinyscript allows array+array and array+element concatenation
function __ts_add(obj1, obj2) { ... }
```

**Every `+` expression in generated code is wrapped as `__ts_add(left, right)`** to preserve Tinyscript's array concatenation semantics.

### `for` Loop Transpilation

- **Numeric `for (i = start to end)`:** emits `for (var i = start; i <= stop; i = i + 1)` with a bidirectional step condition when a step is specified
- **Iterable `for (x in array)`:** emits a C-style for loop with an internal `__ts_i` counter, guarded by an `Array.isArray()` check

### Identifier Shadowing

`getName(Identifier id)` calls `TinyscriptModelUtil.isShadowing(id)` to detect name collisions between scopes. Shadowed identifiers are emitted with a fully-qualified name (scope path joined with `_`) to prevent JavaScript variable collisions.

## 13. Deployment Modes

### REPL

**Entry point:** `de.mkbauer.tinyscript.repl/src/.../TinyscriptReplMain.java`

Uses JLine for readline-style input with history. Sandboxing is **off** (full access to `fs` and other built-ins). Each input line is evaluated in a persistent `ExecutionContext` so variable state accumulates across lines.

### Web Demo

**Entry point:** `de.mkbauer.tinyscript.webdemo/src/.../TinyscriptWebController.java`

Spring Boot application. REST endpoints:

| Path | Method | Action |
|---|---|---|
| `/execute` | POST | Run Tinyscript; returns output + execution stats |
| `/xcompile` | POST | Transpile to JavaScript; returns JS source |
| `/test` | GET | Run a hardcoded sample program |

The `TinyscriptExecutionService` creates a fresh sandboxed `TinyscriptEngine` per request (stateless). Output is captured via a custom `OutputStream` with an 8 KB limit. Sandboxing is **on**.

### JVM ScriptEngine

**Entry point:** `de.mkbauer.tinyscript.jvmscriptengine/src/.../JvmScriptEngineRunner.java`

Implements the standard `javax.script.ScriptEngine` interface. Internally:
1. Parses and validates Tinyscript
2. Uses `TinyscriptGenerator` to transpile to JavaScript
3. Executes the JavaScript via GraalJS (falling back to Nashorn if unavailable)

This is the integration point for embedding Tinyscript in a JVM application via the standard scripting API.

### Eclipse Plugin

**Modules:** `de.mkbauer.tinyscript.ide` + `de.mkbauer.tinyscript.ui`

Most of the IDE module is auto-generated by Xtext from the grammar. The `ui` module adds the Eclipse editor integration (syntax coloring, outline view, etc.). These do not contain hand-written interpreter logic — they delegate to the core module.

## 14. Key Design Decisions

**Why integer switch for dispatch?**
EMF classifier IDs are compile-time constants, making the switch statement a direct jump table. This is significantly faster than `instanceof` chains for a hot loop that visits every AST node in a script.

**Why exception-based return values?**
`TSReturnValue` as a thrown exception cleanly unwinds arbitrarily nested recursive `execute()` calls without threading a "return flag" through every handler. The same technique could be extended for `break`/`continue` (currently handled differently).

**Why `TSValue.UNDEFINED` and `TSValue.NULL` as identity singletons?**
Both have `value == null` internally. Using object identity (`this == TSValue.UNDEFINED`) rather than a tag field avoids an extra field on every `TSValue` instance and makes the common `undefined`/`null` checks a single pointer comparison.

**Why `GlobalExecutionContext` uses a `TSObject` instead of a `HashMap`?**
The global object is directly accessible from script code as `this` at global scope. Using a `TSObject` means global variables are automatically properties of `this`, matching ECMAScript semantics and allowing `Object.keys(this)` to enumerate globals.

**Why a `WeakHashMap` for object tracking?**
The resource monitor needs to track live object count and memory without preventing GC. A `WeakHashMap` automatically removes entries when the tracked object is collected, and the callback (`onExpunge`) decrements the tracked memory total in real time.

## 15. Test Architecture

### Two test modules

**`de.mkbauer.tinyscript.tests`** (OSGi, `eclipse-test-plugin` packaging)

Uses `tycho-surefire-plugin` to run JUnit 4 tests inside a full OSGi container. Requires the target platform. Tests use `@RunWith(XtextRunner.class)` and `@InjectWith(TinyscriptInjectorProvider.class)` to get a properly wired Guice injector with all Xtext services. Test resources (`.ts` files) live in `resources/`.

This module is the right place for:
- Tests that cover grammar, validation, or scoping details
- Tests that verify cross-reference resolution and IDE-level features

**`de.mkbauer.tinyscript.generator.tests`** (plain `jar`, standard Maven Surefire)

No OSGi. Bootstrapped via `TinyscriptStandaloneSetup.createInjectorAndDoEMFRegistration()`. Runs in < 5 s. Uses `node` (via Apache Commons Exec) as a differential oracle for the transpiler.

This module is the right place for:
- Interpreter correctness tests (one `assertEquals` per language feature)
- Transpiler differential tests (interpreter output vs node output)
- Integration tests for complete programs

### Two-level test design in `generator.tests`

| Level | Class | Assertion | On failure |
|---|---|---|---|
| 1a — interpreter | `InterpreterReferenceTest` | `assertEquals(expected, interpreterOutput(tsCode))` | Exact wrong value; points to buggy `TinyscriptEngine` visitor |
| 1b — differential | `TranspilerTest` | `assertSameOutput(tsCode)` | Source + generated JS + both stdouts shown; paste JS into `node` to reproduce |
| 2 — integration | `TranspilerVerifyTest` | `assertTranspilerCorrectFromFile(filename)` | Generated JS + node stderr shown |

Tests run alphabetically: `InterpreterReferenceTest` → `TranspilerTest` → `TranspilerVerifyTest`.

### When to run each module

After any change to `TinyscriptEngine` or `TinyscriptGenerator`, run the generator tests first:

```bash
mvn test -pl de.mkbauer.tinyscript.generator.tests
```

Then run the full OSGi test suite before merging:

```bash
mvn clean test
```
